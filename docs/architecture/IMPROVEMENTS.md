# Architectural Improvement Plan

## 1. Overview

The current architecture, while functional, is a script-based pipeline where data is passed between stages via the filesystem. This can lead to challenges in maintainability, scalability, and error handling.

This document outlines a plan to refactor the scanner into a more robust, modular, and scalable system. The core goal is to evolve the project from a collection of scripts into a cohesive Python application, without losing the functionality of the original design.

The key pillars of this architectural evolution are:

1.  **Introduce a Centralized Data Model:** Establish a single, authoritative data structure to represent the state of a scan, ensuring consistency and decoupling the pipeline stages.
2.  **Refactor into a Modular, Object-Oriented Design:** Reorganize the codebase from monolithic scripts into a well-defined Python module with clear separation of concerns.
3.  **Implement a More Robust Data Flow:** Transition from a filesystem-based data flow to an in-memory model, with options for future database-backed state management.
4.  **Enhance Error Handling and Reporting:** Build a more resilient pipeline that can handle failures gracefully and provide detailed error reports.

These improvements are designed to be implemented incrementally, allowing for a smooth transition from the current architecture to the new one.

## 2. Key Architectural Changes

### From Filesystem to In-Memory Data Flow

-   **Current:** `pipeline.sh` calls Python scripts in sequence. Each script reads data from files generated by the previous one (e.g., `audit.py` processes folders created by `fetch.py`).
-   **Proposed:** A single application entry point (`main.py`) will orchestrate the pipeline. Data will be held in memory within a structured data model and passed between components. This eliminates tight coupling to the filesystem and makes the data flow explicit.

### From Scripts to a Cohesive Application

-   **Current:** The logic is spread across `fetch.py`, `audit.py`, and `summarize.py`, with orchestration in `pipeline.sh`.
-   **Proposed:** The logic will be consolidated into a new `scanner` Python module. The `pipeline.sh` script will be simplified to just an entry point that launches the Python application, which will manage the entire workflow internally.

### From Implicit to Explicit Data Structures

-   **Current:** The "shape" of the data is implicitly defined by how files are named and structured within the `scan_...` directories.
-   **Proposed:** A formal data model will be defined in `scanner/models.py`. Classes like `ScanRun`, `AppScan`, and `AuditResult` will provide a clear, explicit, and type-hinted structure for all data processed by the pipeline.

## 3. Benefits of the New Architecture

-   **Maintainability:** With a clear separation of concerns and a well-defined data model, the codebase will be significantly easier to understand, debug, and modify.
-   **Scalability:** The modular design and improved data flow will make it easier to scale the pipeline, for instance, by distributing work across multiple machines or using a more robust data store.
-   **Extensibility:** Adding new features, such as a new type of security check or a different output format (e.g., HTML reports), will be a matter of adding a new module or class, rather than modifying monolithic scripts.
-   **Testability:** The logic will be encapsulated in classes and functions that can be easily unit-tested, leading to a more reliable and robust application.
-   **Robustness:** Centralized state management and improved error handling will make the pipeline more resilient to failures.
